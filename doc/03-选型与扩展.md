# 选型与扩展

## 1. 当前选型

- 当前基础框架统一使用 **LangChain `create_agent`**。
- 不再默认同时维护一套独立的 LangGraph 工作流入口。
- 原则是：**能用一个 Agent 解决，就不要先上图编排**。

这样做的原因：

- 入口只有一个，理解和调试成本最低。
- 工具、提示词、模型配置都只维护一份。
- 当前项目还没有复杂分支、循环、人审、多角色协作，提前引入图会增加心智负担。

## 2. 什么时候再引入 LangGraph

满足下面任一条件，再单独引入 LangGraph 更合适：

- 明确需要多节点分支或循环。
- 需要在中间步骤做人审、暂停、恢复。
- 需要把一个任务拆成多个稳定阶段，每个阶段状态都要显式保存。
- 需要多个子 agent 协同，而不是单 agent 调工具。

在这些需求出现前，保持单一 Agent 入口更符合“简洁快速、方便易懂”的目标。

## 3. 当前扩展建议

- **新工具**：优先通过 `agents/tools/` 扩能力，而不是先加新编排层。
- **流式输出**：直接在现有 agent 上加 stream，再由 API 层转 SSE 或 WebSocket。
- **复杂流程**：确认需求稳定后，再把复杂部分单独抽成 LangGraph，而不是把整个项目都切成图。
